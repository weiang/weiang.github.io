Linux和Windows NT进程

发表于 2016-10-22

进程概述

开发者编写、编译和链接的程序以磁盘文件系统中文件的方式存储在计算机磁盘上。这种文件的格式是预先定义好的，能被操作系统或操作系统Shell识别为可执行文件。

当程序运行时，程序被实例化在内存中，占用存储数据结构用的内存、文件描述符等系统资源，提供至少一个定义目前状态和进程后续必要操作的执行线程。目前正在运行的程序或进程，在运行状态下对独占处理器，可以使用文件系统中的文件，直接或间接访问系统中的物理设备。

进程有两种运行模式：用户模式或系统模式（也称kernel模式）。进程可以在这两种模式间切换，两种模式可能是同一进程的不同阶段（phase）。包括大多数应用进程在内的进程默认以用户模式运行，操作系统为他们提供相互独立的运行环境，以保证运行在同一机器上的不同进程之间不互相干扰。用户模式下的进程通过系统调用，产生异常（或错误）或发生中断（系统时钟）等方式切换为kernel模式，这是kernel代表进程执行相关操作。任意时刻，进程在自己的上下文中运行，kernel在正在运行的进程的上下文中运行。如下图所示：



kernel模式下的进程是特权的，能够访问所有的计算机资源（如所有可用的内存）而不受用户模式下进程的诸多限制。通过硬件强制特权规则来区别进程的用户模式和kernel模式，可以保护计算机系统不受不当损坏和由于不良行为的用户进程导致失败的影响。

操作系统间进程的基本概念相似，但各自以不同的方式实现进程管理。

  • Linux
    Linux上特权服务在一个整体的kernel上实现。kernel提供操作计算机所必需的核心服务。
    Linux kernel还有一些可动态加载的模块来补充核心功能，如增加一个新的文件系统。由于kernel以一个实体来执行一系列的任务，所以被称为macrokernel架构。

  • Windows NT
    Windows NT操作系统基于分层操作系统的衍生物，使用客户端/服务器的microkernel架构。microkernel架构的核心kernel进程尽可能高效的执行最基本的任务。
    与microkernel相关联的是一系列特权进程，称为the Executive。它们在各自分开的进程子系统中运行，包括专用的进程管理器，对象管理器，虚拟内存管理器，以提供成熟的服务来支持进程的构建和资源的管理。

操作系统本身主要在kernel模式下运行。用户模式下的进程需要请求的某些服务只能由运行于kernel模式下的特权代码提供。这种情况下，它们调用系统函数，触发硬件trap，导致进程上下文切换到kernel模式，由可信的操作系统函数执行需要的操作，然后上下文切换到用户模式。

每一个用户进程都在自己的地址空间（32位处理器3GB）中运行，独立于其他进程。从单个进程的角度来看，它独占处理器，如在操作系统的控制下，它运行于独立于其他进程的属于自己的虚拟机。

虚拟内存

虚拟内存是一种运行超过计算机物理内存地址空间可容纳进程的机制。

每一个等待处理器的进程都会被分配自己的虚拟内存区域，这个区域定义了进程执行任务可以访问的逻辑地址。鉴于这个虚拟内存区域非常大（通常由处理器的地址位数和操作系统支持的最大进程数决定），每个进程都会被分配非常大的逻辑地址空间。

虚拟内存管理器的任务是在需要是保证活动进程和它们希望访问的区域被映射到物理内存。这是通过将指定的区域换入换成物理内存的方式来实现的。交换(swap)涉及整个进程的替换，翻页(page)涉及用另外一个进程的某个页来替换本进程的某页。交换和翻页是计算密集型和时间消耗型任务，要注意最小化它们带来的额外成本。一系列利用相关代码局部性设计的算法和操作优化（必要时才执行某些操作，如内存读取和复制时采用copy-on-write，lazy-paging，demand-paging等）来最小化成本。

进程的虚拟内存中包含的数据和代码可能有多个来源。可执行代码可以以共享库的方式在进程间共享，由于这些区域是只读的，所以不太可能被破坏。进程可以在它们运行时分配和链接虚拟内存。

许多操作系统使用了一下内存管理技术，其中包括Linux和Windows NT的一些：

  • 基于页的保护机制
    每一个虚拟页都有一系列的标记来决定用户模式或kernel模式下的访问权限。
  • 按需分页/延迟读取
    虚拟内存只在进程读取它时才被加载到物理内存中。
  • 用户和kernel模式操作
    kernel模式下进程的内存访问无限制，用户模式下进程只能访问自己的内存。
  • 映射文件
    利用磁盘文件用来暂存从物理内存换出的页面来拓展内存。
  • 写时复制内存
    当两个进程对一个代码区域进行只读访问时，虚拟内存管理器并不立即复制该区域。只有当对该区域的执行写操作时，才复制。
  • 共享内存
    通过调用特权操作可以将一块内存空间映射到多于一个进程的地址空间中。
  • 内存锁定
    为了保证关键页面不被换出内存，将该页面锁住，虚拟内存管理器将不会把它从内存中删除。
  • Windows NT中：基于对象的内存保护
    任何进程尝试打开内存区域句柄或映射它时，NT security reference将检查访问权限。

进程和线程

进程，线程简介

计算机是完成特定任务的工具，我们把特定的任务称为task。最早的计算机是单task系统，需要手动装载，启动task的程序，所有的资源由该task独占，一个task完成后才能启动下一个task。操作系统的一个重要功能是管理task，为task分配系统资源。进程是操作系统分配资源和调度task的基本单位。

进程是一个具有一定独特功能的程序在一个数据集合上的一次动态执行。进程由代码块（静态，二进制），数据集，进程控制块（动态）三部分组成。进程概念的引入是为了便于理解操作系统对task管理。进程是操作系统为task分配系统资源的基本单位，也是task调度的基本单位。

线程是CPU计算流的抽象，一般认为它由程序计数器（PC），栈和寄存器组构成。从操作系统的角度来看，线程使资源和计算流分离，多个线程可以共享系统资源，实现了系统资源（内存，文件，信号）的高效利用。同时，这种共享也实现了计算流之间更高效的交互，实现更高的并行和并发度。

C和C++中的lvalue

发表于 2015-05-07   |   分类于 Technology

lvalue的来源

lvalue的提出主要是为了判断赋值表达式的使用是否符合规范。关于lvalue，通常的理解为‘left value’，意指用在赋值表达式左侧的表达式。K&R中是这样描述的：

The name lvalue comes from the assignment expression E1 = E2 in which the left operand E1 must be an lvalue expression.

这个定义只是指出赋值表达式的左侧应该是一个lvalue，并没有给出什么样的表达式才是lvalue，于是K&R中给出了lvalue的准确定义：

An lvalue is an expression referring to an object.

这个定义可以用来判断哪些表达式是lvalue。即表达式表示的是一个在占有内存的对象时，这个表达式就是lvalue。

那么问题来了，c语言中变量分为两类，一类值不可被修改（如常量变量，临时变量），另一类值可被修改。毫无疑问，常量变量是lvalue，但是它不能被赋值修改，它不能位于赋值表达式的左侧。所以K&R中关于lvalue用于赋值表达式左侧的论述是不完全的，只有拥有可修改内存的lvalue才能用于赋值表达式的左侧。

为了避免歧义，C99建议将lvalue理解为locator value，这样更加符合K&R中关于lvalue的定义。同时将用于赋值表达式左侧的对象表达式成为modifiable lvalue，将用于赋值表达式右侧的值称为rvalue，意味表达式的值(value of an expression)。

Singleton Pattern
Ensure a class only has one instance and provide a global point of access to it.

为什么需要Singleton模式

定义类的唯一目的就是可以用它作为模板来创建对象,而利用Singleton模式实现的类只创建和维护一个对象,这是不是有违定义类的初衷?其实不然,多数情况下类存在的目的是以其为模板创建该类对象的实例,而某些特定的引用场合下,如果一个类存在多个实例对象可能会对系统产生负面影响,甚至导致错误的发生.如在针对操作系统线程池的应用中,如果系统中存在多个线程池显然是错误的.当然有时系统中存在多个对象并不发生错误,但可能占用过多的系统资源导致资源利用率不高,这些都不是我们期望发生的,所以在面向对象的编程中引入了Singleton模式来解决这些问题.

什么是变量作用域(variable scope)

作用域定义了一个变量在程序中的可访问范围。Ruby中有四种类型的变量作用域：局部(local),全局(global),实例(instance)和类(class)。通过在变量名前加上不同的特殊字符来声明不同的变量类型。如下表：

变量名前缀 作用域
$ 全局变量
@ 实例变量
[a-z]或_ 局部变量
[A-Z] 常量
@@ 类变量

除此之外，Ruby还有两个不能被赋值的伪变量(pseudo-variables)，nil和self。nil用来表示未初始化的变量，self用来引用正在执行的对象。
